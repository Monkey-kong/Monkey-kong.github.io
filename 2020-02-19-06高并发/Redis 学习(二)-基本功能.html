    1. Redis 管道（Pipelining）

    1.1 一次执行多条语句减少 RTT

    使用管道，可以有效减少 RTT (Round Trip Time - 往返时间) 

    // 如果没有 redis 客户端，只要能和 redis 建立一个 socket 连接，也是可以执行的
yum instll nc
nc localhost 6379
set key1 aaa
echo -e "set k2 99\n incr k2\n get k2" | nc localhost 6379

    1.2 Redis 从文件中批量插入数据

    // 创建文件，输入命令
vim d1.txt
// redis-cli中只支持dos格式的换行符 \r\n；需要转码，不过我没转也成功了
yum install unix2dos
unix2dos d1.txt
// 执行导入：cat 读取文件内容；redis-cli 发送文件到 redis 执行
cat d1.txt | redis-cli --pipe
cat d1.txt | redis-cli -p 6380 -h 192.168.1.166 --pipe

    2. Redis 发布/订阅（Pub/Sub）

    1. 接收不到订阅之前的消息。

    2. 发布/订阅与 key 所在空间没有关系，包括不同数据库编码。 发布在 db 2，订阅可以在 db 1 

    help @pubsub
// 订阅
subscribe ooxx
// 模糊匹配订阅（可能导致收到多条消息）
psubscribe ooxx.*
// 发布
publish ooxx hello
// 取消订阅
unsubscribe
// 模糊匹配取消订阅
punsubscribe
// 查看一些统计信息
pubsub channels/numpat/numsub

    2.1 发布/订阅应用场景

    如 QQ 消息，别人发消息需要实时查看，也可查看历史数据。

    方案一：

    发布消息时需要做三件事情：

    1. 将发布到 redis 指定管道
    2. 将消息存到 redis 的 sorted set（日期作为分值）
    3. 将消息发布到 kafka 指定 topic

    接收消息时：

    1. 接收实时消息：订阅 redis 指定管道
    2. 近期消息：查询 redis sorted set
    3. 历史消息：查询 mysql

    方案二：

    发布消息时：

    1. 发布消息到 redis 服务指定管道
    2. redis 实例2 订阅消息存 zset（日期作为分值）
    3. service 端订阅消息发 kafka

    接收消息时：不变

    3. Reids 事务

    set k1 aaa
// 放弃事务
discard
// 执行 multi 标记后的所有命令
exec
// 标记事务开始
multi
// 取消监视，有啥用？和 discard 有啥区别
unwatch
// watch 可为 redis 事务提供 CAS 行为。使用 check-and-set 操作实现乐观锁
// 当 EXEC 被调用时， 不管事务是否成功执行， 对所有键的监视都会被取消
watch

    4.1 事务的理解

    例子：两个 redis 客户端，client1 想查询 k1，client2 想删除 k1 如何保证事务？

    先执行 get，还是先执行 del，取决于 exec1 先到达还是 exec2 先到达，先到先执行。注意这里两个事务都执行了。加了 watch 就不一样了。

    如果 exec2 先到达，执行 exec1 时会报错。所以一般 client1 执行前会 watch k1，加了watch 后，如果执行 exec1 时，k1 有改动，则该事务内的所有指令不会执行。

    4.2 错误发生在 exec 前后的情况

    // 语法错误，放弃事务
multi
set k1 aaa
get k1 bbb // 无法入队
exec

// 执行时错误，只是错误的命令不执行
multi
set k1 aaa
lpop k1   // 成功入队，但是类型错误
exec

    4.3 为什么 Redis 不支持回滚

    Redis 在事务失败时不进行回滚，而是继续执行余下的命令 。

    优点：

    • Redis 命令只会因为错误的语法而失败（并且这些问题不能在入队时发现），或是命令用在了错误类型的键上面：这也就是说，从实用性的角度来说，失败的命令是由编程错误造成的。Redis 官方认为，这种情况和你本来想 set k1 3，结果错误的写成 set k1 4 是一个道理，是没法回滚的。
    • 因为无需对回滚进行支持，所以 Redis 的内部可以保持简单且快速

    6. Redis 作为数据库和作为缓存的区别

    6.1 作为缓存

    1. 缓存数据“不重要”，不是全量数据
    2. Redis 里的数据应该随着业务的变化而变化，只保留热点数据，因为内存大小是有限的，也就是瓶颈。

    6.2 如何实现数据随着业务变化？

    1. 业务逻辑：根据业务逻辑来确定 key 的有效期

      set k1 aaa ex 20
// 查看过期时间
ttl k1
// 更新过期时间（重新设置，记录 session 可能会用到）
expire k1 60
// 读操作，过期时间不会变，还是在继续减少
get k1
// 写操作，重置过期时间。ttl k1 ==> -1
set k1 bbb
// 当前时间戳
time
// 定时（时间戳）
expireat k1
// 移除过期时间
PERSIST k1

    2. 业务运转：内存是有限的，随着访问的变化，应该淘汰掉冷数据。

      1. maxmemeory 设置最大内存

      2. maxmemory-policy 设置回收策略。LFU：使用次数；LRU：多久没有使用。

    6.3 Redis 如何淘汰过期的 key？

    1. 被动访问判定。访问到某个 key，发现其过期，然后删除。

    2. 周期轮询时间判定（增量），每秒执行 10 次。

      1. 测试随机的20个keys进行相关过期检测。
      2. 删除所有已经过期的keys。
      3. 如果有多于25%的keys过期，重复步奏1。

      目的，稍微牺牲下内存，但是保住了 redis 性能。

    6.4 Redis 作为数据库

    1. 数据绝对不能丢
    2. 内存数据库都有掉电易失的问题，所以需要持久化

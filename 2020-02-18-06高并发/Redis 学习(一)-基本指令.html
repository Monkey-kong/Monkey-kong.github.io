    1. Redis 前世今生

    1.0 一点常识

    • 磁盘寻址速度 ms 级别，带宽 G 或 M 级别
    • 内存寻址速度 ns 级别，比磁盘寻址快了 10 万倍，带宽极大
    • 磁盘有磁道和扇区，一扇区默认 512 Byte。

    1.1 最初数据是存在文件中

    比如 linux 使用 grep、awk 来检索数据，随着文件的变大，速度会变慢。因为是全量检索，会遇到磁盘寻址和吞吐的瓶颈。数据库使用分治来解决，Redis 作为缓存，存的不是全量数据，而是热数据。

    1.2 关系型数据库

    数据库使用多个 datapage 存储表的数据，每个 datapage 4k （可以调整）大小，把数据分开存储，然后可以有索引来检索这些数据，索引也是存储在 datapage 中。索引使用内存里的 B+ 树来管理，B+ 树树干是在内存中，索引放在 B+ 树的叶子上，sql 语句中 where 条件只要命中索引， B+ 树就沿着树干找到索引的叶子，然后把叶子 datapage 读到内存解析，然后知道解析（放到内存中）哪些数据 datapage。

    结构图如下：

    问题：关系型数据库建表时为什么必须给出 schema ?

    答案：因为存储数据时，倾向于行级存储。就是说即使这个表有十个字段，有一行数据只有一个字段有值，存储时也会开辟十个字段的 datapage。这样的好处是后边增删改这行数据就不用做偏移了，直接重写就行。

    问题：数据库表很大，会导致性能下降？

    答案：如果表有索引，增删改需要维护索引，性能会变慢。查询则分两种情况：1. 一个或少量查询依然会很快。 2. 并发大的时候会受到磁盘带宽的影响导致速度变慢

    问题：什么是 4k 对齐？

    答案：正常来说，磁盘一个扇区 512 Byte。所谓 4k 对齐就是指按照 4k 来划分小格子，索引的体量大小变小了，操作系统读取的时候，不论读取多大的数据都是按照 4k 读取。注意磁盘格子的大小不一定是 4k，这取决于上层应用。比如上层应用主要用于存储视频，音频等，4k 可能太小了，浪费了磁盘寻址时间，反而得不偿失。

    缺点：

    • 数据是存在磁盘的文件里。随着文件变大，速度会变慢，因为硬盘 I/O 成为瓶颈。
    • 磁盘寻址慢
    • 随着数据量增大，索引成本变大

    1.3 关系型内存数据库缺点

    如果直接使用关系型内存数据库，速度确实快，比如 SAP 公司的 HANA 数据库就是关系型内存数据库，但是价格太贵，一般买不起。但是磁盘数据库又太慢了。

    所以出现了折中方案：key-value 型内存数据库： memcached、Redis

    1.4 key-value 型内存数据库 Redis

    Redis 是一个开源（BSD许可）的，内存中的数据结构存储系统，它可以用作数据库、缓存和消息中间件。

    Redis 支持多种类型的数据结构，如字符串、散列、列表、集合、有序集合与范围查询，bitmaps、hyperloglogs 和地理空间索引半径查询。

    Redis 内置了复制，LUA 脚本，LRU 驱动事件，事务和不同级别的磁盘持久化，并通过 Redis 哨兵和自动分区提供高可用性。

    1.5 Redis 和 memcached 对比

    在 Redis 之前，有memcached，也是 key-value 型数据库，但是它的 value 没有类型的概念，所以更多人选择使用 Redis。

    问题： memcached 的 value 也可以用 json 来存储复杂的数据结构，为啥会选择 Redis？

    答案：因为获取数据成本不同，比如从一个 list 类型的 value 中取一个元素：

    • memcached 需要返回 value 所有的数据到客户端，就是整个 list json 数据。会导致两个问题：1. server 端网卡 IO 限制。2. 客户端需要自己实现代码去解码 json。
    • Redis 其实类型不是很重要，它本身是二进制安全的，重要的是 Redis 中对每种类型都有自己的方法。比如根据 index、lpop 等。避免的上面的两个问题（返回需要的数据，降低 I/O 消耗；计算向数据移动，计算主要在 Redis server 端完成）。

    问题：Redis 为什么不用 sql 查询，而是 key-value？

    答案：因为 Redis 设计的初衷不是存储全量数据，而是存储部分数据。部分数据无法维护类似关系型数据库中的约束、范式，所以还不如面向数据本身，把场景简单化。

    2. 卸载/安装 Redis

    2.1 安装 Redis

    // 1. 检查是否已经安装 wget
rpm -qa|grep wget
// 2. 如果没有安装则安装 wget
yum install wget
// 3. 下载 redis 源码到家目录
cd ~
mkdir soft
cd soft
wget http://download.redis.io/releases/redis-5.0.5.tar.gz
// 4. 解压源码
tar xf redis-5.0.5.tar.gz
// 5. 阅读 README.md 文件
cd redis--5.0.5
vi README.md
// 6. 编译源码
make
// 7. 如果提示 cc:command not found；否则跳过
yum install gcc
make distclean
make
// 8. 发现源码 src 目录已经生成了可执行文件
// 9. 安装 redis 到指定目录
make install PREFIX=/opt/alvin/redis5
// 10. 添加 path 环境变量
vi /etc/profile
  export REDIS_HOME=/opt/alvin/redis5
  export PATH=$PATH:$REDIS_HOME/bin
// 11. 刷新环境变量
source /etc/profile
// 12. 将 redis 安装为服务
cd utils
// 可执行一次或者多次
// a) 一个物理机中可以有多个 redis 实例（进程），通过端口号区分
// b) 可执行程序就一份在目录，但是内存中未来的多个实例需要各自的配置文件，持久化目录等
// c) redis 服务已经自动启动
../../install_server.sh
// 13. 查看服务状态
service redis_6379 status
// 14. 查看 redis 服务进程
ps -ef|grep redis

    2.2 卸载 Redis

    // 1. 检查是否已经安装 redis
whereis redis-server
// 2. 如果安装了，停止 redis 服务
service redis_6379 stop
// 3. 删除 redis 服务
// 不删除也无影响
// 4. 删除安装文件
rm -rf redis5
// 5. 删除源码
rm -rf redis-5.0.5
rm -rf redis-5.0.5.tar.gz

    4. Redis 五种 value 类型

    4.1 string

    // =================通用类开始====================
// 查看帮助
redis-cli -help
// 客服端连接 -n 表示连那个库，一共有16个库
redis-cli -p 6379 -n 0
// 不带raw：redis 只会识别 ascii 码，超出 ascii 直接显示 16 进制
// 带了raw：会触发编码集的格式化。发现了当前的十六进制符合当前编码字符集的规则，就会转换
redis-cli --raw
// 切换库
select 8
// 如何学习命令
help
// 查看 value 类型
type k1
// type 还是 string，因为 set 命令是 string 组的
set k1 99
object help
// 查看 value 编码 int
object encoding k1
// =================通用类结束====================
// =================字符串类开始===================
// nx（只能新增）：k1 不存在的时候才操作
set k1 ooxx nx
// xx（只能更新）：k2 存在的时候才操作
set k2 hello xx
setex k1 5 aaa
psetex k1 5000 aaa
// mset、mget 设置、获取多条
mset k3 a k4 b
mget k3 k4
// msetnx(只能新增)
msetnx k1 a k2 b
msetnx k2 c k3 d
// 只找到 k1、k2 说明 msetnx 是原子性的
mget k1 k2 k3
// append 追加
append k1 " world"
// getrange 获取索引范围内字节
getrange k1 6 10
// 有正向索引和反向索引
getrange k1 6 -1
getrange k1 0 -1
// 覆盖指定位置开始的字符串，能覆盖多少就覆盖多少
setrange k1 6 mashibing
// 字符串长度
strlen k1
// =================字符串类结束===================
// =================数值类开始=====================
// 加1
incr k1
// 加22
incrby k1 22
// 减1
decr k1
// 减22
decrby k1 22
// 加0.5
incrbyfloat k1 0.5
// =================数值类结束=====================
// =================bitmap类开始==================
// 面向字节有索引，二进制位也有索引
// offset 的 1 是指位索引的第二个位置设置为 1
// 注意 setbit 是位级别 set
// strlen为1 @
setbit k1 1 1
// strlen为1 A
setbit k1 7 1
// strlen为2 A@
setbit k1 9 1
getbit k1 0
// 指定范围直接内，查询第一次出现指定位的位索引：1
bitpos k1 1 0 0
// 9
bitpos k1 1 1 1
// 1：第一个字节到第二个字节，1第一次出现1的位索引
bitpos k1 1 0 1
// 4：第一个字节到第二个字节1出现的次数
bitcount k1 0 1
// k1 k2 按位与的结果存到 andkey
bitop and andkey k1 k2
// 按位或
bitop or orkey k1 k2
// =================bitmap类结束==================

    4.1.1 string 类型应用

    1. 分布式锁： string 类型的 setnx 的作用是“当 key 不存在时，设值并返回 1，当 key 已经存在时，不设值并返回 0”，“判断 key 是否存在”和“设值”两个操作是原子性地执行的，因此可以用 string 类型作为分布式锁，返回 1 表示获得锁，返回 0 表示没有获得锁。
    2. 存储对象：对象转为 json，存入 Redis。

    4.1.2 数值类型应用

    计数器：抢购、秒杀、点赞、评论数。规避并发下，对数据库的事务操作，完全由 reids 内存操作代替。

    4.1.3 bitmap 类应用

    1. 有一个用户系统，需要统计用户登录天数，且窗口（日期）随机，怎么统计？

      使用用户 id 作为 key，value 为 bitmap，位索引表示日期，位值为 1 表示登录了。

      // alvin 第二天登录了
setbit alvin 1 1
// alvin 第八天登录了
setbit alvin 7 1
// alvin 第 365 天登录了
setbit alvin 364 1
// 仅占用 46 个字节
strlen alvin
// 今年最后 16 天 alvin 登录次数
bitcount alvin -2 -1
// 问题：最后 15 天怎么统计呢？窗口期只能是 8 的倍数？或者索引全部乘以 8，这样空间又提升了 8 倍。或者 getbit 一个一个取，这样访问量又上去了，考虑使用管道或许可行。

    2. 京东 618 做活动，送礼物，大库应该备货多少？即活跃用户统计。

      以日期作为 key，value 为 bitmap，位索引表示员工编号，位值为 1 表示登录了。

      // 一月一日一号员工登录
setbit 20190101 1 1
// 一月二日一号员工登录
setbit 20190102 1 1
// 一月二日七号员工登录
setbit 20190102 7 1
// 一日、二日这两天多少人登录了？
bitop or destkey 20190101 20190102
bitcount destkey 0 -1
// 问题：实际情况员工或者用户 id 不一定是 1 开始排序。

    4.1.4 bitfield 命令理解

    // BITFIELD key 
// [GET type offset] 
// [SET type offset value] 
// [INCRBY type offset increment] 
// [OVERFLOW WRAP|SAT|FAIL]
set key1 a
// 97,a ASCII 码就是 97
bitfield key1 get i8 0

// 有符号 8 位，表数范围（-128~127）
bitfield key1 set i8 0 127
// 127
bitfield key1 get i8 0
// 溢出默认 WARP
bitfield key1 set i8 0 128
// -128
bitfield key1 get i8 0
// 设置溢出 sat
bitfield key1 overflow sat set i8 0 128
// 127
bitfield key1 get i8 0
// 设置溢出 fail
bitfield key1 overflow fail set i8 0 128
// 127(没有变化)
bitfield key1 get i8 0

// 无符号 8 位，表数范围（0~255）
bitfield key1 set u8 0 255
// 255
bitfield key1 get u8 0
// 溢出默认 WARP
bitfield key1 set u8 0 257
// 1
bitfield key1 get u8 0

    5.1 scan 命令理解

    SCAN 命令及其相关的 SSCAN, HSCAN 和 ZSCAN 命令都用于增量迭代一个集合元素。

    他们都支持增量式迭代，它们每次执行都只会返回少量元素，所以这些命令可以用于生产环境，而不会出现像 KEYS 或者 SMEMBERS 命令带来的可能会阻塞服务器的问题。相对的有一个缺点，就是 scan 在迭代过程中，集合元素可能会被修改，对返回的值无法提供完全的保证。

    scan 是基于游标的迭代器。每次迭代需要上次迭代返回的游标。游标为 0 表示一个新的迭代；服务器返回 0 时表示迭代结束。

    问题：

    • 同一个元素可能会被返回多次
    • 如果一个元素是在迭代过程中被添加到数据集的， 又或者是在迭代过程中从数据集中被删除的， 那么这个元素可能会被返回， 也可能不会。
    • SCAN增量式迭代命令并不保证每次执行都返回某个给定数量的元素,甚至可能会返回零个元素， 但只要命令返回的游标不是 0 ， 应用程序就不应该将迭代视作结束。

    0. MySQL 的基本架构

    1. 性能监控

    1.1 profiles、profile

    查看 sql 消耗的时间、cpu、io 等信息。

    # 设置后，show profiles 才有显示
set profiling = 1;
# 业务查询语句
select * from mylock;
# 查看 sql 查询总耗时
show profiles;
# 查看详细耗时
show profile;
# 查看指定 sql
show profile for query 2;
# cpu 信息也展示出来
show profile cpu;
# 展示所有
show profile all;
# 官网地址：SQL Statements -> Database Administration Statements -> SHOW Statements

    问题：java 执行的 sql 好像监控不到

    1.2 performance_schema

    数据不会持久化。每次重启 mysql 服务后会清空。

    # 查看是否开启
SHOW VARIABLES LIKE 'performance_schema';

# 语句事件记录表。当前语句、历史语句、聚合信息
show tables like '%statement%';

# 等待事件记录表
show tables like '%wait%';

# 事务事件记录表
show tables like '%transaction%';

# 监控文件系统层调用
show tables like '%file%';

# 配置表
show tables like '%setup%';

# 有些 instruments 和 consumers 默认没有启动，需要手动开启
## 比如，开启等待事件监控功能
UPDATE setup_instruments SET ENABLED = 'YES', TIMED = 'YES'where name like 'wait%';
UPDATE setup_consumers SET ENABLED = 'YES'where name like '%wait%';

# 查看相关的一些系统变量
show variables like '%performance_schema%';

    1.3 show processlist

    查看连接的线程个数，观察是否有大量线程处于不正常的状态或者其他不正常的特征。

    2. schema 与数据类型优化

    2.1 数据类型的优化

    长度越小越好

    尽量使用可以正确存储数据的最小数据类型，越小的数据类型占用更少的磁盘空间、内存和 CPU 缓存，并且处理时需要的 CPU 周期更少。

    类型越简单越好

    1. 整型比字符操作代价更低
    2. 使用自建类型来存储日期和时间，而不是字符串
    3. 使用整型来存储 IP 地址，而不是字符串

    尽量避免 null

    ​ null 列对 mysql 来说很难优化，因为可为 null 的列使得索引、索引统计和值比较都更加复杂。

    实现细节

    • 整数类型： TINYINT，SMALLINT，MEDIUMINT，INT，BIGINT分别使用8，16，24，32，64位存储空间。 尽量使用满足需求的最小数据类型 。

      -- 1 2 3 4 8 = 18 字节
-- 长度没有限制作用
create table int_test(a tinyint(2), b smallint(2), c mediumint, d int, e bigint);
tinyint:-128(10000000)~127(01111111)

    • 字符和字符串类型

      • char：长度固定；最长 255 字符；适用于身份证号、手机号、摘要算法等定长字符串。字符和字节的关系：gbk-1个字符2个字节；utf8-1个字符3个字节
      • varchar：长度可变；可以设置最大长度；最大空间 65535 个字节，因为 MySQL 要求一个行定义长度不能超多 65535 字节（不包含 text、blob）；适用于长度变化频繁、多字节字符（汉字、特殊字符）。
      -- char、varchar 指定长度后，不能超过该长度
-- char 不指定长度，默认为 1
-- varchar 必须指定长度，否则建表失败
create table char_test(a char(4), b char, c varchar(4), d varchar(20));
      • varchar 到底可以定义多长？

      最大 65535 字节，字符集为 utf8，每个字符 3 字节，所以最大 21845 字符。

      mysql> create table varchar_test(name varchar(65536));
ERROR 1074 (42000): Column length too big for column 'name' (max = 21845); use BLOB or TEXT instead
mysql> show variables like '%char%';
+--------------------------+--------------------------------------------------+
| Variable_name            | Value                                            |
+--------------------------+--------------------------------------------------+
| character_set_client     | utf8                    |
| character_set_connection | utf8                    |
| character_set_database   | utf8                    |
| character_set_filesystem | binary                  |
| character_set_results    | utf8                    |
| character_set_server     | utf8                    |
| character_set_system     | utf8
+--------------------------+--------------------------------------------------+

    • datetime 和 timestamp

      • datetime：占用 8 个字节；存储方式与时区无关；时间范围（100000101-99991231）精确到毫秒；
      • timestamp：占用 4 个字节；存储方式依赖数据库时区，时区变化，数据会变；时间范围（19700101-20380119）；精确到毫秒；更适合于 跨时区的业务。

    • date：占用 3 个字节；时间范围（10000101-99991231）

      create table date_test(a datetime, a1 datetime(6),b timestamp, b1 timestamp(3) null default null,c date);
insert into date_test values(now(),now(6),now(),'20200808121212.123',now());

mysql> select * from date_test\G
*************************** 1. row ***************************
 a: 2020-08-29 21:45:55
a1: 2020-08-29 21:45:55.862122
 b: 2020-08-29 21:45:55
b1: 2020-08-08 12:12:12.123
 c: 2020-08-29

    • 使用枚举代替字符串类型

      • mysql 会根据枚举值把数据压缩到一个或两个字节中，内部会将每个值在列表中的位置保存为整数。并且在表的 .frm 文件中保存“数字-字符串”映射关系的查找表。
      CREATE TABLE `enum_test`  (
  `id` int(6) NOT NULL AUTO_INCREMENT,
  `e` enum('男','女') CHARACTER SET utf8 COLLATE utf8_general_ci NOT NULL,
  PRIMARY KEY (`id`) USING BTREE
);

INSERT INTO `enum_test`(e) VALUES(1),(2);
INSERT INTO `enum_test`(e) VALUES('男'),('女');
INSERT INTO `enum_test`(e) VALUES(1),('1'),('女');

select * from enum_test;
select e+1 from enum_test;

    • 使用 int(32) unsigned 来存 ip 地址，因为 ip 地址本来就是 32 位

      create table ip_test(host int(32) unsigned);
insert into ip_test values(inet_aton('192.168.238.68'));
select inet_ntoa(host) from ip_test;

    2.2 合理使用范式和反范式

    范式优点

    • 更新通常比反范式快
    • 很少或者没有冗余数据
    • 范式化的数据比较小，可以放在内存中，操作比较快

    范式缺点

    • 通常需要进行关联

    反范式优点

    • 所有的数据都在同一张表中，可以避免关联
    • 可以设计有效的索引
    • 空间换时间的思想

    反范式缺点

    • 数据冗余较多
    • 删除数据时会造成表有些有用的信息丢失，完全反范式所有内容存在一张表，删除用户消息导致用户信息都丢了，所以一般混合使用，user 表和 message 表都存用户类型字段。
    • 需要确保数据的一致性不会被破坏，确保更新时冗余字段也被更新

    注意

    • 在企业上很难做到严格意义上的范式或者反范式，一般混合使用。

      • 比如用户表和用户信息表均冗余用户类型字段。这样既方便了消息的查询也避免了没有消息时，用户类型信息的丢失
      • 另一个从父表冗余数据到子表的理由是排序的需要
      • 缓存衍生值也是有用的。比如维护一个 num_messages 字段到 user 表，记录每个用户发了多少消息。或者说订单表存一个订单价格字段，而不用每次从订单商品中去计算价格

    2.3 主键的选择

    1. 代理主键：与业务无关，无意义的数字序列
    2. 自然主键：事物属性中的自然唯一标识

    推荐使用代理主键

    • 不与业务耦合，更容易维护
    • 通用的主键策略：减少需要编写的源码数量，减少系统的总体拥有成本

    2.4 字符集的选择

    字符集直接决定了数据在 MySQL 中的存储编码方式，由于同样的内容使用不同字符集表示所占用的空间大小会有较大的差异，所以通过使用合适的字符集，可以帮助我们尽可能减少数据量，进而减少 IO 操作次数。

    1. 纯拉丁字符能表示的内容，没必要选择 latin1 之外的其他字符编码，因为这会节省大量的存储空间
    2. 如果可以确定不需要存放多种语言，就没必要使用 utf8 或者其他 unicode 字符类型，造成大量的存储空间浪费
    3. MySQL 的字符集类型可以精确到字段

    2.5 存储引擎的选择

    MyISAM InnoDB
    索引类型 非聚簇索引 聚簇索引
    支持事务 否 是
    支持表锁 是 是
    支持行锁 否 是
    支持外键 否 是
    支持全文索引 是 是（5.6之后）
    适合操作类型 大量 select 大量 insert、delete、update

    2.6 适当的数据冗余

    即适当的反范式。

    • 被频繁引用，且只能通过 join 2张（或者更多）大表2的方式才能得到的独立小字段。
    • 这样的场景由于每次 join 仅仅只是为了取得某个小字段的值，join 到的记录又大，会造成大量不必要的 IO，完全可以通过空间换时间的方式来优化。不过，冗余的同时需要确保数据的一致性不会遭到破坏，确保更新的同时冗余字段也被更新。

    2.7 适当的拆分

    当我们的表中存在类似于 TEXT 或者是很大的 VARCHAR 类型的大字段的时候，如果我们大部分访问这张表的时候都不需要这个字段，我们就该义无反顾的将其拆分到另外的独立表中，以减少常用数据所占用的存储空间。这样做的一个明显好处就是每个数据块中可以存储的数据条数可以大大增加，既减少物理 IO 次数，也能大大提高内存中的缓存命中率。

    3. 二进制

    1、byte 占用一个字节， 8 位，对于计算机来说表数范围为 0000 0000 ~ 1111 1111

    2、最高位表示符号位

    3、计算机用补码表数

    4、正数和 0 的补码 = 源码

    5、负数的补码 = 其对应正数按位取反 + 1

    int binVal4 =0B10000000 00000000 00000000 00000011  // 的值为？
// 1. int 类型 32 位，高位为 1，所以是个负数
// 2. 该负数对应正数按位取反的值 = 0B10000000 00000000 00000000 00000011 - 1
    = 0B10000000 00000000 00000000 00000010
// 3. 该负数对应正数 = OB01111111 11111111 11111111 11111101      
// 4. 该负数 = OB11111111 11111111 11111111 11111101
    // byte 类型，-127 的补码？
// 1. 127 的源码 = 0111 1111
// 2. 127 的源码按位取反 = 1000 0000
// 3. 127 的源码按位取反 + 1 = 1000 0001

// byte 类型，-128 的补码？
// 1. 128 的源码 = 1000 0000
// 2. 128 的源码按位取反 = 0111 1111
// 3. 128 的源码按位取反 + 1 = 1000 0000

// byte 类型，-129 的补码？
// 1. 129 的源码 = 1000 0001
// 2. 129 的源码按位取反 = 0111 1110
// 3. 129 的源码按位取反 + 1 = 0111 1111

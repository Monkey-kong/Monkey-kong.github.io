    第 10 章 异常处理

    10.1 异常概述

    异常机制可以使程序中的异常处理代码和正常业务代码分离,保证程序代码更加优雅,并可以提高程序的健壮性。

    如果没有异常机制：

    if("用户输入包含逗号之外的其他非数字字符"){
    alert "坐标只能是数值"
    goto retry
} else if("用户输入不包含逗号") {
    alert "应使用逗号分隔两个坐标"
    goto retry
} else if("用户输入的坐标值超出有效范围") {
    alert "坐标应位与棋盘坐标之内"
    goto retry
} else if("用户输入坐标已有棋子") {
    alert "只能在没有棋子的地方下棋"
    goto retry
} else {
    // 业务代码
    ...
}

    上面代码缺点：

    • 无法穷举所有的异常情况，所以代码总是不够健壮
    • 错误处理代码和业务实现代码混杂，影响程序可读性，不利于维护

    如果有一种强大的机制来解决上面的问题，例如：

    if("用户输入不合法") {
    alert "输入不合法"
    goto retry
} else {
    // 业务实现代码
    ...
}

    10.2 异常处理机制

    Java 异常机制，当程序运行出现意外情况时，系统会自动生成一个 Exception 对象来通知程序，从而实现业务代码和错误处理代码分离。

    10.2.1 使用 try… catch 捕获异常

    如果用异常处理机制：

    try {
    // 业务实现代码
    ...
} catch (Exception e) {
    alert "输入不合法"
    goto retry
}

    10.2.2 异常类的继承体系

    Java 提供了丰富的异常类，这些异常类之间有严格的继承关系。

    Error 错误一般无法处理，将导致程序中断。所以无需 catch、throw、throws Error。

    所有父类异常的 catch 块应该排在子类异常 catch 块后面，即先处理小异常，再处理大异常。否则编译出错。

    10.2.3 Java 7 新增的多异常捕获

    Java 7 以前，一个 catch 块只能处理一种类型异常；Java 7 开始可以处理多个。

    • 多个异常类型用竖线(|)隔开
    • 捕获多种异常时，异常变量有隐式的 final 修饰，因此不可修改
    public class MultiExceptionTest {
    public static void main(String[] args) {
        try {
            int a = Integer.parseInt(args[0]);
            int b = Integer.parseInt(args[1]);
            int c = a / b;
            System.out.println("您输入的两个数相除的结果是：" + c);
        } catch (IndexOutOfBoundsException|NumberFormatException
                |ArithmeticException ie){
            System.out.println("程序发生了数组越界、数字格式异常、算术异常之一");
            // 多异常有 final 修饰，编译报错
            // ie = new ArithmeticException("test");
        } catch (Exception e) {
            System.out.println("未知异常");
            // 正常
            e = new RuntimeException("test");
        }
    }
}

    10.2.5 使用 finally 回收资源

    物理资源（数据库连接、网络连接和磁盘文件）需要显示回收。垃圾回收机制不会回收物理资源，只会回收堆内存中对象所占用的内存。

    finally 保证 try 中打开的物理资源可以得到回收，即使 try 块出现异常或者 使用了 return。

    try 块必须有，catch 和 finally 至少有一个。

    public class FinallyTest {
    public static void main(String[] args) {
        FileInputStream fis = null;
        try {
            fis = new FileInputStream("a.txt");
        } catch (IOException ioe) {
            System.out.println(ioe.getMessage());
            // return 语句强制返回
            // return ;
            // exit 退出虚拟机。JDK 9 以前会导致不会执行 finally，
            // 到了 jdk 12，即使退出虚拟机，也会 执行 finally
             System.exit(1);
        } finally {
            // 关闭磁盘文件，回收资源
            if(fis != null) {
                try{
                    fis.close();
                } catch (IOException ioe) {
                    ioe.printStackTrace();
                }
            }
            System.out.println("执行了finally");
        }
    }
}

    通常情况下，不要在 finally 中使用 return 或 throw 等导致方法终于的语句，一旦在finally 中食用，将导致 try、catch 中的 return、throw 语句失效。

    public class FinallyFlowTest {
    public static void main(String[] args) {
        boolean a = test();
        System.out.println(a);
    }
    public static boolean test() {
        try {
            // 因为 finally 中包含return，下面语句失效
            throw new RuntimeException("aaa");
            // return true;
        } finally {
            return false;
        }
    }
}

    因为 try、catch 中遇到 return、throw 并不会立即终止方法，而是先找 finally，如果有finally 会先执行 finally，如果 finally 中包含 return、throw 会直接导致方法结束。

    10.2.6 异常处理的嵌套

    异常处理代码可以放在任何可执行性代码的地方，没有规定嵌套层数。但是一般不会超过两层。

    10.2.7 Java 9 增强的自动关闭资源的 try 语句

    为了保证 try 语句可以正常关闭资源，这些资源实现类必须实现 AutoCloseable 或 Closeable 接口。

    相当于包含了隐式的 finally 块。

    Java 7 开始

    public class AutoCloseTest {
    public static void main(String[] args) throws IOException {
        try(
                // 声明、初始化两个可关闭的资源
                BufferedReader br = new BufferedReader(new FileReader("a.txt"));
                PrintStream ps = new PrintStream(new FileOutputStream("b.txt")))
        {
            // 使用两个资源
            System.out.println(br.readLine());
            ps.println("aaaaaabbbbbb");
        }
    }
}

    Java 9 再次增强

    Java 9 不要求在 try 后的圆括号内声明并创建资源，只需要自动关闭的资源有 final 修饰或者是有效的 final (effectively final) , Java 9 允许将资源变量放在 try 后的圆括号内。

    public class AutoCloseTest2 {
    public static void main(String[] args) throws IOException {
        // 有 final 修饰的资源
        final BufferedReader br = new BufferedReader(new FileReader("a.txt"));
        // 没有显示使用 final，但只要不对该变量重新赋值，该变量就是有效的 final
        PrintStream ps = new PrintStream(new FileOutputStream("b.txt"));
        // 只要将两个资源放在 try 后的圆括号内即可
        try (br;ps) {
            // 使用两个资源
            System.out.println(br.readLine());
            ps.println("aaaaaabbbbbb");
        }
    }
}

    10.3 Checked 异常和 Runtime 异常体系

    Java 的异常被分为两大类：Checked 异常和 Runtime 异常（运行时异常）。所有的 RuntimeException 类及其子类的实例被称为 Runtime 异常；其他的则成为 Checked 异常。

    Java 认为 Checked 异常是可以被处理（修复的），所以必须显示处理 Checkedd 异常，否则编译报错。

    有两种处理方法：

    • catch
    • throws 声明抛出

    Runtime 异常则更加灵活，Runtime 异常无需显示声明抛出，如果程序要捕获，也可以捕获。

    10.3.1 使用 throws 声明抛出异常

    表示当前方法不知道如何处理该异常，抛出去给调用者处理。如果最后 main 方法也不知道如何处理，抛出异常给 JVM，则 JVM 打印堆栈，程序终止。

    10.3.2 方法重写时声明抛出异常的限制

    子类声明抛出的异常类型应该是父类抛出的异常类型的子类或相同；

    子类声明抛出的异常不允许比父类声明抛出的异常多。

    public class OverrideThrows {
    public void test() throws IOException {
        final FileInputStream fis = new FileInputStream("a.txt");
    }
}
class Sub extends OverrideThrows {
    // 子类方法声明抛出了比父类方法更大的异常
    // 所以下面方法出错
//    public void test() throws Exception {
//        super.test();
//    }
}

    由此可以见，使用 Checked 异常至少有两处不便：

    • 对于程序中出现的 Checked 异常，必须要显示的声明抛出或者捕获处理该异常
    • 方法声明中抛出 Checked 异常，导致方法签名与异常耦合。

    所以大部分时候推荐使用 Runtime 异常，而不使用 Checked 异常。尤其是当程序自行抛出异常时。

    10.4 使用 throw 抛出异常

    10.4.1 抛出异常

    public class ThrowTest {
    public static void main(String[] args) {
        try {
            // 调用声明抛出 Checked 异常的方法，要么显示捕获
            // 要么在 main 方法中再次声明抛出
            throwChecked(-3);
        } catch (Exception e) {
            System.out.println(e.getMessage());
        }
        // 调用声明抛出 Runtime 异常的方法既可以显示捕获
        // 也可以不理会
        throwRuntime(3);
    }
    public static void throwChecked(int a) throws Exception {
        if(a>0) {
            // 自行抛出 Exception 异常
            // 该代码必须处于 try 块里，或处于带 throws 声明的方法中
            throw new Exception("throwChecked a 的值大于 0，不符合要求");
        }
    }
    public static void throwRuntime(int a) {
        if(a>0) {
            // 自行抛出 RuntimeException 异常，既可以显示捕获
            // 也可以不理会该异常，把异常交给方法调用者处理
            throw new RuntimeException("throwRuntime a 的值大于 0，不符合要求");
        }
    }
}

    10.4.2 自定义异常类

    public class AuctionException extends Exception {
    // 无参数构造器
    public AuctionException(){}
    // 带一个字符串参数的构造器
    public AuctionException(String msg) {
        super(msg);
    }
}

    如果需要自定义 Runtime 异常，只需将 Exception 基类改为 RuntimeException 基类。

    10.4.3 catch 和 throw 同时使用

    catch 中使用 throw。

    10.4.4 Java 7 增强的 throw 语句

    public class ThrowTest2 {
    public static void main(String[] args)
        // Java 7 以前，必须声明抛出 Exception。
        throws FileNotFoundException
    {
        try{
            new FileOutputStream("a.txt");
        } catch (Exception ex) {
            ex.printStackTrace();
            throw  ex;
        }
    }
}

    10.4.5 异常链

    大型系统通常会分层。一般不可跨层调用。所以当持久层出现 SQLException 时，不应该把这个底层异常直接传到用户界面，一方面不友好，一方面容易暴露出不安全信息。

    通常的做法是，先捕获原始异常，然后抛出一个新的业务异常，新的业务异常中包含了对用户的提示信息，这种处理方式被称为异常转译。

    这种把捕获一个异常然后接着抛出另一个异常，并把原始异常信息保存下来是一种典型的链式处理，就是责任链模式的应用。

    所以自定义异常时，再增加一个构造器，用于支持链式处理：

    // 接收 Throwable 参数的构造器
// 用于封装原始异常，从而实现对异常的链式处理
public AuctionException(Throwable t){
    super(t);
}

    10.5 Java 的异常跟踪栈

    虽然 printStackTrace() 方法可以很方法的用于追踪异常发生的情况，可以用它来调试程序，但在最后发布的程序中，应该避免使用它；而应该对捕获的异常进行适当的处理，不是简单的打印跟踪栈。

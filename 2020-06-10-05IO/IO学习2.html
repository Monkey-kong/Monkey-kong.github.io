    1. 理解 PageCache

    • 系统中有虚拟的线性地址（逻辑地址）和物理地址映射，线性地址可能是连续的，但是指向的真实物理地址可能不连续。通过 CPU 的 MMU（内存管理单元）来管理映射。映射的最小单位是 4KB。

    • page 是随用随分配的，每页大小 4KB。比如当某个程序申请 40KB 内存，系统不会一次性分配 40KB。当程序访问某个地址发现没有该地址映射的 page 时，系统会产生缺页异常，触发软中断，CPU 就会对该地址进行映射，映射后程序再去访问该地址。

    • 多个程序可以同时访问一个物理文件，这时就体现出 page 的好处了，多个程序可以访问同一个 page，通过文件描述符来记录偏移量，而不用每个程序都把物理文件 load 到内存中。这是一种内存优化机制。

    • page 刚分配时是脏页，脏页写入磁盘后，就不是脏页了，这时可以被 LRU、LFU 清理。

    • page cache 的好处是优化 IO 性能，但是异常断电时会有丢失数据的风险。

    # 查询脏页相关配置
sysctl -a | grep dirty

# 程序向系统写入数据时，基本都是先写内存，再写磁盘。写磁盘有触发时机。
# pagecache 大小达到内核可用内存的 10% 时
# 内核后台启动一个线程把 pagecache 中数据写入到磁盘，这时程序不会阻塞
vm.dirty_background_ratio = 10
# 程序在往内核写的时候，分配 page 分配到内存的 20% 时，阻塞程序
vm.dirty_ratio = 20
# 5 秒做一次脏页写回磁盘。单位：1/100秒
vm.dirty_writeback_centisecs = 500
# 30 秒脏页的生命周期，就是脏页可以存多久。单位：1/100秒
vm.dirty_expire_centisecs = 3000

vm.dirty_background_bytes = 0
vm.dirty_bytes = 0

# 修改默认配置
vi /etc/sysctl.conf
# 90% 才开始回写脏页
vm.dirty_background_ratio = 90
# 90% 才阻塞程序
vm.dirty_ratio = 90
# 50 秒才写回脏页 
vm.dirty_writeback_centisecs = 5000
# 脏页可以存在 300 秒
vm.dirty_expire_centisecs = 30000

    画一个图，解释内核 page cache 机制。回写、缺页、脏页、淘汰。

    DMA（协处理器）：如果没有 DMA，内核向磁盘写回 page 时，CPU 需要先把 page 转移到寄存器，然后由寄存器写回磁盘，转移期间 cpu 被占用，无法处理其他事情；如果有 DMA，DMA 负责把 page 写回磁盘，DMA 处理期间，CPU 可以继续工作。

    程序（进程）有缓存 buffer，内核有 page cache 缓存，磁盘也有缓存，都有可能丢数据。

    2. MMAP

    FileChannel.map() 可以得到 MappedByteBuffer。mmap 是一个堆外的和文件映射的 ByteBuffer。

    在使用 MappedByteBuffer 之前，必须使用 out.write() 这样的系统调用，才能让程序的数据进入内核的 pagecache，也就是说曾经必须有用户态和内核态的切换。但是 MappedByteBuffer.put() 也会把程序数据通过映射写入内核的 pagecache，但是没有调用系统调用。

    mmap 的内存映射，依然受内核的 pagecache 体系所约束的。换言之，同时有丢数据的风险。目前 Java 没有能力让你逃离 pagecache，所有的 Java api 都不行。你可以去 github 上找一些 jni 扩展，使用 Linux 内核的 Direct IO（直接 IO），Direct IO 是忽略 Linux 的 pagecache 的，是把 pagecache 交给了程序自己开辟一个字节数组作为 pagecache，动用代码逻辑来会维护一致性/dirty 等等一系列复杂问题。自己实现还是同样可能丢失数据，但是可以自己更加细粒度的控制各个参数，因为 Linux 系统的配置参数是全局的，一改全改。

    /**
 * 文件 IO
 * 1. 测试最基本文件 IO(无 buffer)
 * 2. 测试 buffer文件 IO
 * 3. 测试 NIO Channel、ByteBuffer、MappedByteBuffer
 */
public class OSFileIO {
    static byte[] data = "123456789\n".getBytes();
//    static String path =  "/root/iolearing/io2/out.txt";
    static String path =  "./src/main/java/com/monkeykong/c2/out.txt";
    public static void main(String[] args) throws Exception {
        switch ( args[0]) {
            case "0" :
                testBasicFileIO();
                break;
            case "1":
                testBufferedFileIO();
                break;
            case "2" :
                testRandomAccessFileWrite();
            default:
        }
    }
    /**
     * 测试最基本文件IO(无 buffer)
     */
    public static  void testBasicFileIO() throws Exception {
        File file = new File(path);
        FileOutputStream out = new FileOutputStream(file);
        while(true){
            Thread.sleep(10);
            out.write(data);
        }
    }
    /**
     * 测试 buffer文件 IO
     * 1. 有缓存，默认 8kb 大小，可以通过构造函数自定义缓冲区大小
     * 2. 有 flush 方法，调用 OutputStream 的 write 方法，把缓冲区数据写入内核
     * 3. write 方法是将数据写入缓冲区，缓冲区满了时才将数据写入内核
     * @throws Exception
     */
    public static void testBufferedFileIO() throws Exception {
        File file = new File(path);
        BufferedOutputStream out = new BufferedOutputStream(new FileOutputStream(file));
        while (true) {
            Thread.sleep(10);
            out.write(data);
        }
    }

    /**
     * 测试 NIO(Buffer Channel)
     * @throws Exception
     */
    public static void testRandomAccessFileWrite() throws  Exception {
        Path p = Paths.get(OSFileIO.path);
        if (Files.exists(p)) Files.delete(p);
        // 1. IO，使用普通写，写完文件大小 12 字节
        RandomAccessFile raf = new RandomAccessFile(OSFileIO.path, "rw");
        raf.write("normal write".getBytes());
        System.out.println("====>raf写："+raf.length());
        readFromRaf(raf);
        System.in.read();

        // 2. NIO，使用 mmap 写，文件 4096 字节
        FileChannel rafChannel = raf.getChannel();
        MappedByteBuffer mapBuffer = rafChannel.map(FileChannel.MapMode.READ_WRITE, 0, 4096);
        mapBuffer.put("mmap ByteBuffer write".getBytes());
        System.out.println("====>mmap buffer写："+raf.length());
        readFromBuffer(mapBuffer);
        System.in.read();

        // 3. NIO，使用 channel 写
        ByteBuffer allocateBuffer = ByteBuffer.allocate(8192);
        // ByteBuffer buffer = ByteBuffer.allocateDirect(1024);
        allocateBuffer.put("normal ButeBuffer write".getBytes()).flip();
        rafChannel.position(0);
        rafChannel.write(allocateBuffer);
        System.out.println("====>channel buffer写："+raf.length());
        readFromChannel(rafChannel);
    }

    private static void readFromChannel(FileChannel rafChannel) throws IOException {
        rafChannel.position(0);
        ByteBuffer tmpBuffer = ByteBuffer.allocate(1024);
        while (rafChannel.read(tmpBuffer) > 0) {
            readFromBuffer(tmpBuffer);
        }
    }

    private static void readFromBuffer(ByteBuffer mapBuffer) throws CharacterCodingException {
        mapBuffer.flip();
        CharBuffer mapCharBuffer = StandardCharsets.UTF_8.newDecoder().decode(mapBuffer);
        System.out.println(mapCharBuffer.toString());
        mapBuffer.clear();
    }

    private static void readFromRaf(RandomAccessFile raf) throws IOException {
        raf.seek(0);
        byte[] tmpBuffer = new byte[1024];
        while (raf.read(tmpBuffer) > 0) {
            System.out.println(new String(tmpBuffer, Charset.defaultCharset()));
        }
    }
}

    3. 总结图

    问题：buffer io 为什么快？

    答案：系统调用明显减少，也就是内核态和用户态的切换明显减少。buffer io 每次处理一批字节，普通 io 每次处理提交的字节。buffer 默认大小 8kb，既到达 8kb 的数据后才会调用系统调用，而普通的 io 每次 write 都是调用系统调用。

    问题：epoll 是怎么知道数据到达的？

    答案：中断。

Token 安全性问题、Oauth2

1. Token 安全性问题

1.1 问题

  • 重放攻击
    • 非法 js 读取 header 中的 token，这就可以发送非法请求了

1.2 Token 携带方式

  • http header
  • url 上的 Get 请求
  • Cookies
  • localstorage

https/ssl/tls：数据防篡改、防钓鱼网站

token 不可能防止被盗。？？？

微服务之间没必要走 https，nginx 和 公网之间使用 https 保证数据不会在公网被篡改。

微服务之间要防止家贼。

精通接口安全 Token

token 只能用一次安全吗？

token 用完之后，放入缓存，相当于加入了黑名单。

能否防止 xss 攻击？不能

能否先执行非法 js，再加载 token？不行，非法 js 可以一直跑。

所以怎么做都没法防止非法 js

token 有效期设置短一点

加入人机交互：图形验证码，旧密码，删除（confirm 1+1=?）

读请求，读到后怎么发出去，跨域发送

防止非法 js 进入系统。这些 js 哪里来的，发帖子，表单提交

非法字符转义 URLEncode，可以拦主互联网用户。

怎么防止服务器端？

扫描 服务器 静态资源，数据库表字段 <script src

不管怎么样，一定要记录日志，用于追溯。

综合：web 端防不住

server 下发 app，通过 https

app 没有 xss 攻击。app 装在每个手机，抓包只能抓到自己的数据，抓不到别人数据。

除非 webview 可能存在重放攻击。可以通过本地代理，过滤无效请求。

token 丢了咋办？

转移责任：短音验证码、ip 改变时发出安全提示

人机交互：滑动

代码混淆：加盐、无法反编译，乱码

物理隔离：

放内部人员。需要通过制度约束。

2. oauth2.0

2.1 访问流程

  1. 用户访问资源服务器，发现用户没有权限，跳转登录页
  2. 用户访问认证服务器进行登录认证，得到权限
  3. 认证服务器带着该用户的权限通过重定向地址再次访问资源服务器
  4. 资源服务器发送 authentication code、client id、client secret 到认证服务器获取 access token
  5. 用户登录

中间 authentication 有何作用，为啥要用访问两次认证服务器才拿到 token，第一次直接拿 token 不行吗？

2.2 角色

Resource Owner

资源拥有者。

Resource Server

资源服务器

Client Application

第三方客户端

Authorization Server

认证服务器

2.3 授权类型

anthorization_code

授权码类型，适用于 Web Server Application。

模式为：

  • 客户端先调用/oauth/authorize/进到用户授权界面，用户授权后返回 code
  • 客户端然后根据 code 和 secret 获取 access token

implicit

简化类型，相对于授权码类型少了授权码获取的步骤。

password

密码类型。

client_credentials

客户端类型

2.4 应用场景

  • 单点登录
  • 第三方登录
    • 微信授权：https://developers.weixin.qq.com/doc/oplatform/Mobile_App/WeChat_Login/Development_Guide.html

    1. 聚簇索引和非聚簇索引

    1.1 聚簇索引

    不是索引类型，而是一种数据的存储方式，指的是数据行跟相邻的键值紧凑的存储在一起。

    优点：

    1. 可以把相关的数据保存在一起
    2. 数据访问更快，因为索引和数据保存在同一个树中
    3. 覆盖索引的查询可以直接使用页节点中的主键值

    缺点：

    1. 聚簇数据最大限度的提高了 IO 密集型应用的性能，但是如果数据全部放在内存中，本来就不需要太多的 IO，那么聚簇索引就没有什么优势
    2. 插入的速度严重依赖于插入顺序，按照主键的顺序插入是最快的方式
    3. 更新聚簇索引列的代价很高，因为会强制将每个被更新的行移动到新的位置
    4. 在插入新行、主键被更新导致需要移动行的时候，可能面临页分裂问题。
    5. 聚簇索引可能导致全表扫描变慢，尤其是行比较稀疏，或者由于页分裂导致数据存储不连续的时候。

    1.2 非聚簇索引

    将数据文件和索引文件分开存放。

    MyISAM 索引文件和数据文件分开存放（非聚簇索引）。索引存储在索引文件表名称.MYI中，数据文件存储在数据文件 表名称.MYD中。

    InnoDB 主键索引（聚簇索引），其他索引都称之为辅助索引或者二级索引，但是也是聚簇索引，存的是主键值，而不是数据行地址。InnoDB的数据和索引存储在一个文件表名称.ibd 中。

    2. 索引匹配方式

    2.1 全值匹配

    可以和组合索引中所有列进行匹配。

    create table staffs(
    id int primary key auto_increment,
    name varchar(24) not null default '' comment '姓名',
    age int not null default 0 comment '年龄',
    pos varchar(20) not null default '' comment '职位',
    add_time timestamp not null default current_timestamp comment '入职时间'
  ) charset utf8 comment '员工记录表';
alter table staffs add index idx_nap(name, age, pos);
show index from staffs;
-- 匹配索引
explain select * from staffs where name = 'July' and age = '23' and pos = 'dev';

    2.2 匹配最左前缀

    可以只匹配的组合索引的前面几列。

    
-- 匹配索引 type=ref
explain select * from staffs where name = 'July' and age = '23';
-- 匹配索引 type=ref
explain select * from staffs where name = 'July';
-- 全表扫描 type=all
explain select * from staffs where age = '23';

    2.3 匹配列前缀

    可以匹配某一列的值的开头部分。

    -- 匹配索引 type=range
explain select * from staffs where name like 'J%';
-- 匹配索引 type=range
explain select * from staffs where name like 'J%' and age = 23;
-- 全表扫描 type=1all
explain select * from staffs where name like '%y';

    2.4 匹配范围值

    范围查询也可以匹配索引，但是会中断组合索引。

    -- 匹配索引 type=range
explain select * from staffs where name > 'Mary';
-- age 未走索引 type=range
explain select * from staffs where name > 'Mary' and age = 23;

    2.5 精确匹配某一列并范围匹配另外一列

    可以查询第一列的全部和第二列的部分

    -- 匹配索引 type=range
explain select * from staffs where name = 'July' and age > 25;

    2.6 只访问索引查询-覆盖索引

    3. 索引优化细节

    3.1 索引列不要使用表达式

    当使用索引列进行查询的时候尽量不要使用表达式，把计算放到业务层而不是数据库层。

    select actor_id from actor where actor_id=4;
-- possible_keys:null key:idx_actor_last_name ?
select actor_id from actor where actor_id+1=5;

    3.2 尽量使用主键查询

    尽量使用主键查询，而不是其他索引，因为主键查询不会触发回表查询。

    3.3 使用前缀索引

    有时候需要索引很长的字符串，这会让索引变的大且慢，通常情况下可以使用某个列开始的部分字符串，这样大大的节约索引空间，从而提高索引效率，但这会降低索引的选择性，索引的选择性是指不重复的索引值和数据表记录总数的比值，范围从1/#T到1之间。索引的选择性越高则查询效率越高，因为选择性更高的索引可以让 mysql 在查找的时候过滤掉更多的行。

    一般情况下某个列前缀的选择性也是足够高的，足以满足查询的性能，对应 BLOB,TEXT,VARCHAR 类型的列，必须要使用前缀索引，因为 mysql 不允许索引这些列的完整长度，使用该方法的诀窍在于要选择足够长的前缀以保证较高的选择性，通过又不能太长。

    -- 查询不同长度前缀的选择性
select count(distinct left(city,3))/count(*) as sel3,
count(distinct left(city,4))/count(*) as sel4,
count(distinct left(city,5))/count(*) as sel5,
count(distinct left(city,6))/count(*) as sel6,
count(distinct left(city,7))/count(*) as sel7,
count(distinct left(city,8))/count(*) as sel8 
from citydemo;

-- 选择合适的长度创建前缀索引
alter table citydemo add key(city(7));

    注意：前缀索引是一种能使索引更小更快的有效方法，但是也包含缺点：mysql无法使用前缀索引做order by 和 group by。

    3.4 使用索引扫描来排序

    mysql有两种方式可以生成有序的结果：

    1. 通过排序操作：系统手动排序

    2. 通过索引顺序扫描：利用索引结构排序。使用了索引，多列排序规则一致，没有非索引列排序

      如果 explain 出来的 type 列的值为 index，则说明mysql使用了索引扫描来做排序。extra：Using filesort 表示没有用索引排序。

    --sakila数据库中rental表在rental_date,inventory_id,customer_id上有rental_date的索引
--使用rental_date索引为下面的查询做排序
-- 索引排序
explain select rental_id,staff_id from rental where rental_date='2005-05-25' order by inventory_id,customer_id\G
-- 索引排序
explain select rental_id,staff_id from rental where rental_date='2005-05-25' order by inventory_id desc\G
-- Using filesort 不符合索引匹配规则
-- 这里符合最左匹配规则，但是mysql有自己的优化，如果索引选择性太低，会直接走全表扫描。
-- type为all：EXPLAIN SELECT * FROM rental WHERE rental_date>'2005-05-25';
-- type为range：EXPLAIN SELECT * FROM rental WHERE rental_date<'2005-05-25';
explain select rental_id,staff_id from rental where rental_date>'2005-05-25' order by rental_date,inventory_id\G
*************************** 1. row ***************************

        table: rental

         type: ALL
possible_keys: rental_date
          key: NULL
      key_len: NULL
          ref: NULL

-- Using filesort 两个字段排序不一样
explain select rental_id,staff_id from rental where rental_date='2005-05-25' order by inventory_id desc,customer_id asc\G
-- Using filesort 使用了非索引列排序
explain select rental_id,staff_id from rental where rental_date='2005-05-25' order by inventory_id,staff_id\G

    3.5 union all,in,or 都能够使用索引，但是推荐使用in

    3.6 范围列可以用到索引

    范围条件是：<、<=、>、>=、between

    范围列可以用到索引，但是范围列后面的列无法用到索引，索引最多用于一个范围列

    3.7 强制类型转换会全表扫描

    create table typecast_test(id int primary key auto_increment, phone varchar(11));
alter table typecast_test add index idx_phone(phone);
-- 查看表状态
show table status where name like '%typecast%'\G
alter talbe typecast engine='innodb';
-- type:index
explain select * from typecast_test where phone = 1;
-- type:ref
explain select * from typecast_test where phone = '1';
-- 再加上一个 name varchar(20) 字段
-- type:all type:ref
-- 查看 mysql 数据文件位置
show global variables like `%datadir%`;

    index 和 ref 区别？

    index：只是用到了覆盖索引，并没有用索引过滤数据

    ref：用索引来过滤数据了

    类型要匹配。否则全表扫描了。

    3.8 更新频繁、区分度不高字段不建索引？

    更新可能会变更 B+ 树，可能导致页分裂、页合并等问题。

    区分度不高，即使建立了索引页不能有效的过滤数据。

    一般区分度在 80% 以上的时候就可以建立索引，区分度可以使用 count(distinct(列名))/count(*) 来计算。

    3.9 索引列不为 null

    创建索引的列，不允许为null，否则可能会得到不符合预期的结果。

    3.10 表连接不超过 3 张

    当需要进行表连接的时候，最好不要超过三张表，因为需要 join 的字段，数据类型必须一致。

    能使用limit的时候尽量使用limit。这个索引有啥关系？

    • 索引越多越好

    4. 索引监控

    show status like 'Handler_read%';

    • Handler_read_first：读取索引第一个条目的次数
    • Handler_read_key：通过 index 获取数据的次数
    • Handler_read_last：读取索引最后一个条目的次数
    • Handler_read_next：通过索引读取下一条数据的次数
    • Handler_read_prev：通过索引读取上一条数据的次数
    • Handler_read_rnd：从固定位置读取数据的次数
    • Handler_read_rnd_next：从数据节点读取下一条数据的次数

    5. 问题

    1、插入的速度严重依赖于插入顺序，按照主键的顺序插入是最快的方式

    2、mysql无法使用前缀索引做order by 和 group by

    导入大量数据时，可以先把创建索引属性关掉，先把数据文件拷贝过来，然后再打开索引。

    OLAP 联机分析系统。对历史数据进行分析，以确定未来规划等。

    Cardinality：基数。count(distinct city) 近似值。常用于确定关联哪一个列。

    hyperloglog：https://www.jianshu.com/p/55defda6dcd2

    join 内部实现方式

    A join B 不一定是先读 A 再读 B，MySQL 会进行优化。可以关闭。

    最好 小表 join 大表。

    下面两个 sql 区别

    select * from t1 join t2 on t1.id = t2.id and t1.name = ‘aa’

    select * from t1 join t2 on t1.id = t2.id where t1.name = ‘aa’

TCP 是面向连接的可靠的传输协议。三次握手连接，双方内核级开辟资源。

socket 是一个四元组：CIP CPORT + SIP SPORT，唯一标识一个 socket 连接。也是内核级的，即便你不调用 accept，也会有 socket 连接的建立。

1. 理解 Socket 连接

  1. 服务端启动，服务端监听进程启动
  2. 客户端启动，三次握手完成，四元组建立，资源分配完成，双方已经可以通讯
  3. 客户端发送数据，数据正常传输，存储在服务端 Recv-Q 中
  4. 服务端为四元组分配进程，server.accept()
  5. 服务端读取数据，服务端从 Recv-Q 读取数据

1.1 Socket-服务启动

  • 启动了监听进程，并为之分配 FD。

1.2 Socket-客户端连接

  • 三次握手完成，双方内核 buffer 资源已经分配
  • Socket 四元组建立，但是没有分配进程
  • 没有创建文件描述符（Socket）

1.3 Socket-客户端发送数据

  • 抓包工具监控到数据正常传输

  • 连接信息 Recv-Q 值变大，说明数据已经到达服务端，存在 Recv-Q

  • 服务端进程文件描述符信息均不变

1.4 Socket-服务端读取数据

  • 服务端成功接收到客户端发送的数据

  • 抓包工具没有变化

说明数据传输只传输一次。服务端直接从 socket 连接的内核内存中取数据。

  • tcp 连接已经被分配给指定进程

  • 服务端进程创建了文件描述符（socket）

问题：一个客户端和一个服务端可以有超过 655535 个连接吗？

答案：可以的。一台主机可以配置多个 IP：ifconfig eth0:11 192.168.238.66/24 总之只要四元组不同就可以建立连接。

2. 窗口和拥塞

2.1 什么是窗口？

TCP 协议是按照数据包来发送数据，如果传输的数据比较大，肯定会拆分为多个 MTU 进行传输。如果发送一个确认一个，就没有窗口的概念了，但是这样比较慢。所以在三次握手时，可以协商一个窗口大小，这样就可以一次发送多个数据包，两边的窗口可能都不一样，但在发送数据时都会告诉对方剩余的窗口大小，这样对象就可以根据剩余窗口大小不停发包，减少了很多阻塞等待和确认包的过程，这就是所谓的窗口机制，它解决了拥塞问题。

  • MTU：数据包整体大小，和网卡相关联，ifconfig 可以查看。
  • MSS：数据包数据内容大小（1500-20报文头-20端口号 = 1460）。

2.2 什么是拥塞？

拥塞就是如果对方窗口被填满了，那么回复的数据包中会告诉对方没有余量了，这时候客户端就会阻塞自己，不发数据包了。等到服务端处理掉一些包后，再补一个包告诉客户端有窗口余量了，这个时候客户端再发数据包，这就是所谓的拥塞控制，就是既要提高性能，又要保证不要把对方法爆了。

如果服务端还没有接收连接，客户端就不停的发送数据，这时内核中接收队列是有限的，如果队列满了，后边的数据直接被内核丢弃。其实没有丢弃，好像是客户端拥塞了。

4. 服务端配置

4.1 BACK_LOG

server.bind(new InetSocketAddress(9090), BACK_LOG);

问题：Socket 连接的使用和连接建立与否没有必然关系。那么内核会不会一直积攒连接？赞多少连接？

答案：可以通过 BACK_LOG 配置最多可以有多少个没有分配进程（没有 accept）的连接。比如配置 BACK_LOG 为 2，最多有三个未接收的连接，第四个客户端三次握手连接时连接不上。

4.2 KeepAlive

client.setKeepAlive(CLI_KEEPALIVE);

socket 连接建立之后，只要双方均未主动关闭连接，那这个连接就是会一直保持的，就是持久的连接。keepalive只是为了防止连接的双方发生意外而通知不到对方，导致一方还持有连接，占用资源。

TCP 如果双方建立了连接，很久都没有交换数据，不能确定对方是否还是活跃的。keepalive 开启后会开启心跳，如果没有收到心跳，可以把连接剔除。

10. BIO

任何 IO 模型都会有如下三个步骤：

  • socket=fd3

  • bind(fd3,8090)

  • listen(fd3)

以上三个调用完成后，netstat -natp 才能看到 0.0.0.0:8090 0.0.0.0:* LISTEN

BIO 在哪里阻塞？

  • accept(fd3, –>得到 fd5 // 阻塞

  • recv(fd5, –> // 又有一个阻塞 （通过多线程处理，clone）

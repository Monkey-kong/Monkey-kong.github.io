10.2.5 使用 finally 回收资源

物理资源（数据库连接、网络连接和磁盘文件）需要显示回收。垃圾回收机制不会回收物理资源，只会回收堆内存中对象所占用的内存。

finally 保证 try 中打开的物理资源可以得到回收，即使 try 块出现异常或者 使用了 return。

try 块必须有，catch 和 finally 至少有一个。

通常情况下，不要在 finally 中使用 return 或 throw 等导致方法终于的语句，一旦在finally 中食用，将导致 try、catch 中的 return、throw 语句失效。

因为 try、catch 中遇到 return、throw 并不会立即终止方法，而是先找 finally，如果有finally 会先执行 finally，如果 finally 中包含 return、throw 会直接导致方法结束。

10.2.6 异常处理的嵌套

异常处理代码可以放在任何可执行性代码的地方，没有规定嵌套层数。但是一般不会超过两层。

10.3 Checked 异常和 Runtime 异常体系

Java 的异常被分为两大类：Checked 异常和 Runtime 异常（运行时异常）。所有的 RuntimeException 类及其子类的实例被称为 Runtime 异常；其他的则成为 Checked 异常。

Java 认为 Checked 异常是可以被处理（修复的），所以必须显示处理 Checkedd 异常，否则编译报错。

有两种处理方法：

  • catch
  • throws 声明抛出

Runtime 异常则更加灵活，Runtime 异常无需显示声明抛出，如果程序要捕获，也可以捕获。

10.3.1 使用 throws 声明抛出异常

表示当前方法不知道如何处理该异常，抛出去给调用者处理。如果最后 main 方法也不知道如何处理，抛出异常给 JVM，则 JVM 打印堆栈，程序终止。

10.3.2 方法重写时声明抛出异常的限制

子类声明抛出的异常类型应该是父类抛出的异常类型的子类或相同；

子类声明抛出的异常不允许比父类声明抛出的异常多。

由此可以见，使用 Checked 异常至少有两处不便：

  • 对于程序中出现的 Checked 异常，必须要显示的声明抛出或者捕获处理该异常
  • 方法声明中抛出 Checked 异常，导致方法签名与异常耦合。

所以大部分时候推荐使用 Runtime 异常，而不使用 Checked 异常。尤其是当程序自行抛出异常时。

10.4.5 异常链

大型系统通常会分层。一般不可跨层调用。所以当持久层出现 SQLException 时，不应该把这个底层异常直接传到用户界面，一方面不友好，一方面容易暴露出不安全信息。

通常的做法是，先捕获原始异常，然后抛出一个新的业务异常，新的业务异常中包含了对用户的提示信息，这种处理方式被称为异常转译。

这种把捕获一个异常然后接着抛出另一个异常，并把原始异常信息保存下来是一种典型的链式处理，就是责任链模式的应用。
